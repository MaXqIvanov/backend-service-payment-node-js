# Сервис обработки платежей (Payment Processing Service)

Проект представляет собой изолированный финансовый шлюз для обработки счетов (invoices) и безопасного приема статусов оплаты через криптографически защищенные вебхуки.

## 🚀 Архитектурные особенности и стек

- **Node.js (v24+)** + **Express** + **TypeScript**.
- **MongoDB (Mongoose)** — основное хранилище счетов с оптимизированными составными индексами.
- **Redis** — высокопроизвидительный кэш в оперативной памяти для атомарного контроля уникальности запросов.
- **Jest** — юнит-тестирование API с полной изоляцией сред через хуки запуска.
- **Swagger UI (OpenAPI 3.0)** — интерактивная автогенерируемая документация, доступная "из коробки".

---

## 🔒 Реализованные FinTech-стандарты надежности

1. **Точность денежных расчетов (Anti-Float Corruption):**
   Все расчеты комиссий вынесены на математическую библиотеку `big.js`. На уровне СУБД MongoDB данные строго сохраняются в формате `Decimal128`.

2. **Защита от Replay-атак (Атак повторения):**
   Реализовано скользящее временное окно на основе заголовка `X-Timestamp`. Запросы, отправленные более 15 минут назад, автоматически отклоняются сервером.

3. **Гарантированная идемпотентность (Защита от двойных списаний):**
   - На уровне сети: Заголовок `X-Nonce` проверяется через Redis с TTL, равным временному окну (15 минут). Повторные запросы с тем же nonce блокируются на уровне middleware (`400 Bad Request`).
   - На уровне базы данных (Race Conditions): Смена статуса счета с `pending` на `paid`/`failed` производится через атомарную операцию MongoDB `findOneAndUpdate` со строгой фильтрацией по текущему статусу `{ status: 'pending' }`. Это исключает состояние гонки, даже если два идентичных запроса пройдут сквозь кэш параллельно.

4. **Аутентификация запросов (HMAC-SHA256):**
   Запросы от платежной системы проверяются по заголовку `X-Signature`, содержащему криптографический хеш от сырого тела запроса, подписанный общим секретным ключом `WEBHOOK_SECRET`.

5. **Безопасность конфигурации (Fail-Fast подход):**
   При старте сервера модуль `bootstrap.ts` производит "предполетную" инспекцию среды. Если критические переменные окружения отсутствуют, процесс аварийно завершается до поднятия сетевых портов.

## 🛠 Варианты запуска проекта

### Вариант 1: Полное развертывание через Docker Compose (Рекомендуемый)

В корне проекта подготовлена конфигурация `docker-compose.yml` версии `3.8`. Она автоматически собирает Node.js-приложение из `Dockerfile` и поднимает изолированные контейнеры баз данных MongoDB и Redis в общей виртуальной сети.

**Конфигурация окружения внутри контейнеров:**

- Порт сервера: `3000`
- Подключение к БД: `mongodb://mongodb:27017/payments`
- Подключение к кэшу: `redis://redis:6379`
- Секретный ключ подписи: `super_secret_key@@@@`

**Команда для запуска:**

```bash
docker compose up --build
```

Все необходимые тома (`mongo_data`, `redis_data`) для персистентности данных будут созданы автоматически.

### Вариант 2: Локальный запуск (Без Docker)

1. Убедитесь, что на вашем компьютере запущены локальные инстанции MongoDB и Redis.
2. Скопируйте шаблон переменных окружения:
   ```bash
   cp .env.example .env
   ```
3. Установите зависимости проекта через Yarn:
   ```bash
   yarn install
   ```
4. Запустите сервер в режиме разработки:
   ```bash
   yarn dev
   ```
